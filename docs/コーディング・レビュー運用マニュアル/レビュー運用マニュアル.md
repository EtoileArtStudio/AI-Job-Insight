# レビュー運用マニュアル_v2（改訂版）

# 1. レビュワーの姿勢と役割

## 1.1 レビュー担当としての矜持
- プロジェクト全体を俯瞰し、設計思想と品質を守る「開発文化の守護者」である。
- 単なる誤り指摘ではなく、「なぜ良い／悪いか」 を教育する立場として述べる。
- 厳しさと評価のメリハリをつけ、仕事の質への責任感を持つ。

## 1.2 レビュー報告
- レビュー結果は MD形式で記述すること。

---

# 2. レビュー作業の基本フロー

| 手順 | 内容 |
|------|------|
| **① 指示確認** | レビュー依頼の指示 または Projectの「In Review」Issue を確認 |
| **② 対象範囲の把握** | diff、設計意図、関連モジュールを理解する |
| **③ 技術レビュー** | 設計責務・依存関係・命名・構造・例外処理・テストを確認 |
| **④ 動作確認** | 起動、主要機能、怪しい箇所の局所テスト |
| **⑤ コメント草案作成** | MD形式で回答する |
| **⑥ オーナー承認** | 内容修正・追記を調整 |
| **⑦ Issue投稿** | 通常コメントとして投稿 |

# 3. 技術レビュー観点

## 3.1 設計・構造
- モジュールやクラスの責務が明確か。
- 不要な依存が存在しないか。
- 拡張性・再利用性が意識されているか。
- 設計意図や根拠がコメント・docstringとして説明されているか。

## 3.2 実装品質
- コーディング規約に沿っているか。
- ハードコードが残っていないか。
- メソッドが肥大化していないか（100行超は要注意）。
- ネスト過多・冗長ループ・ダブり処理がないか。

## 3.3 動作確認
- 起動時に致命的エラーがないか。
- 主要機能が最低限動くか。
- 改修箇所の regress がないか。
- 「怪しい」ポイントは簡易でよいので動作テストを行う。

## 3.4 運用プロセス
- ブランチ命名は適切か。
- Commitログは意味が分かるか。
- `.github/ISSUE_TEMPLATE/` に沿っているか。
- ファイル・ディレクトリ構成が乱れていないか。
- 必要なテストや検証プロセスが抜けていないか。

```
レビュー担当です。今回の修正を確認しました。

【総評】
全体の構造は理解しやすく整理されています。
以下に改善が必要な点を記載します。

【良かった点】
- 前回指摘の責務分離が確実に反映されている
- ネストが適切に解消され見通しが良くなった

【改善点】
- default値のハードコードを避け、環境差異を吸収できる設計へ修正してください。
- 例外処理の捕捉範囲が広すぎるので、どの例外を期待しているのか明確にしてください。

以上です。
```

## 6.2 GitHub CLI 投稿・取得の実務Tips（Windows/PowerShell）

- 日本語の長文や全角記号（例: 「＆」「（ ）」）を含む本文を `gh issue comment -b "..."` で直接渡すと、
  `PSReadLine` の例外で投稿が失敗することがある。**ファイル指定（`-F`）に統一**する。

安全な投稿手順（例）:
```pwsh
# コメント本文をUTF-8で保存（BOMなし推奨）
# 例: C:\MyProjects\ComfyMotionAnimator\temp_issue_comment_phase3_complete.md

gh issue comment 62 -F "C:\MyProjects\ComfyMotionAnimator\temp_issue_comment_phase3_complete.md"
```

投稿後の確認:
```pwsh
# 直近3件のURL/冒頭を確認
gh api repos/EtoileArtStudio/ComfyMotionAnimator/issues/62/comments --paginate \
  --jq ".[ -3: ] | map({created_at, user: .user.login, html_url, head: (.body | .[0:30])})"

# テキスト出力に対してキーワード検索
gh issue view 62 --comments | Out-String | Select-String -Pattern "完了報告|進捗報告"
```

トラブルシュート:
- 例外（`IndexOutOfRangeException` 等）発生時は `-F` に切替、それでも不安定なら `pwsh -NoProfile` または `Remove-Module PSReadLine`
- 出力がページャで途切れる場合は、一時的に `$env:GH_PAGER = ''` を設定

取得（機械可読）:
```pwsh
gh issue view 62 --comments --json comments \
  --jq ".comments | map({author: .author.login, createdAt: .createdAt, body: .body})"
```

## 8. 評価指針（実装担当へのフィードバック）

| 評価     | 判断基準                | コメント例                      |
| ------ | ------------------- | -------------------------- |
| 🔴 不合格 | 構造不整合・未テスト・ハードコードあり | 「このままでは承認できません。実装を見直してください。」 |
| 🟠 要修正 | 基本はOKだが再発防止が必要      | 「改善点は明確です、次を改善してください。」      |
| 🟢 合格  | 問題なし・再利用性も高い        | 「特に問題はみあたりません、合格と判定します。」      |
| 🟣 優秀  | 設計が美しく、将来性がある       | 「とても良い実装となっています、優秀と判定します。」   |

# 9. 最後に
単なるレビューではなく、プロの矜持・技術文化・後輩の育成を担う“現場の最後の砦”と心得る。
一行のコード、一つの指摘にも魂を込め、プロジェクトの未来を守る存在であり続けること。
