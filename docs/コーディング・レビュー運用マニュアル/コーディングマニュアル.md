# 🧭 コーディングマニュアル

## 1. 役割と姿勢

1. **専門家としての誇りを持つ。**
   　アルゴリズム、設計、保守性において手抜きは一切しない。
2. **先輩レビューア（Codex）への敬意を忘れない。**
   　レビュー指摘は真摯に受け止め、改善を通じて成長する。
3. **要件定義書や仕様書の意図を最優先する。**
   　Issue やコメントに記された文脈・方針を常に参照する。

---

## 2. 作業開始前のルール

1. **該当プロジェクトの Issue 状況を確認。**
   　- 「In Progress」状態の Issue を最優先で対応する。
   　- 具体的な指示がある場合は、その Issue の本文または最新コメントを読み、作業内容を決定する。
2. **ブランチ管理を徹底する。**
   　- 作業対象ブランチが存在しない場合は、新しいブランチを作成してよい。
   　- 命名例：`feature/<機能名>`、`refactor/<対象モジュール>`、`bugfix/<問題箇所>`
3. **Pull Request 作成時は、Issue 番号を必ず関連付ける。**
   　例：`Fixes #23` または `Refs #23`

---

## 3. 設計・実装方針

### 3.1 コード品質

1. **安易なやっつけ仕事をしない。**
   　既存のクラスや関数が再利用可能であれば、それを優先する。
2. **疎結合・高凝集を意識する。**
   　依存関係は最小限に抑え、モジュール単位で独立してテスト可能な設計にする。
3. **モジュール分割を怠らない。**
   　1ファイルあたりおおむね600行以内を目安とする。
   　1000行を超える場合は責務分割を検討する。
4. **実行環境に配慮する。**
   ソースコード内に絵文字などの特殊Unicode文字を使用しない。
   実行環境やエディタ設定による文字コード問題を防ぐため、
   ソースコードは基本的にASCIIまたは通常の日本語コメントに限定する。
   
5. **メインメソッドを用意し、単体実行可能に。**
   　CLI などから直接動作確認ができる設計を目指す。

### 3.2 設定とデータ管理

1. **ハードコードを禁止。**
   　ファイルパス・パラメータ・閾値などは設定ファイル（例：`config.yaml`, `.env`）から読み込む。
2. **定数は設定ファイルを使用して管理する。**
     設定ファイルは config.yaml を基本とする。
     環境依存情報（APIキーなど）は .env を使用する。
     プロジェクト固有の設定ファイルなどを使用してパラメータの外だし、柔軟性の担保を行う。
 
### 3.3 コメントと可読性

1. **日本語コメントを明確に記述する。**
   　コードの意図・背景・副作用を簡潔に説明する。
2. **関数／クラスには docstring を必ず付与。**
   　- 処理概要
   　- 引数 (`Args:`)
   　- 戻り値 (`Returns:`)
   　- 例外 (`Raises:`)

### 3.4 変更履歴の記録

1. **対応Issueの変更内容をCHANGELOG.mdに記録する。**
   　- 機能追加、変更、修正を行った場合は必ず `CHANGELOG.md` の `[Unreleased]` セクションに追記する
   　- 記録形式は [Keep a Changelog](https://keepachangelog.com/ja/1.0.0/) に準拠する
   　- カテゴリは `Added` / `Changed` / `Fixed` / `Deprecated` / `Removed` / `Security` から選択
   　- 変更内容には Issue番号を含める（例: (#37)）
2. **CHANGELOGエントリの記述例：**
   ```markdown
   ### Added
   - [module/file.py] 新機能の説明 (#Issue番号)
   
   ### Fixed
   - [module/file.py] バグ修正の説明 (#Issue番号)
   ```
3. **リリース時の対応：**
   　- リリース担当者が `[Unreleased]` セクションの内容を正式バージョンセクションに移動する
   　- 詳細は `docs/release_process.md` および `docs/versioning_policy.md` を参照

## 4. テスト方針

1. **安易にテスト用スクリプトを量産しない。**
   　既存モジュールを直接呼び出す形式でテストする。
2. **pytest など統一されたテスト基盤を使用。**
   　テストファイルは `tests/` ディレクトリ配下に配置。
3. **テスト完了後は必ず結果を記録。**
   　テスト結果は/docs/以下の適切なディレクトリに記録として残す。

## 5. Git 運用ルール

1. **作業ブランチで開発 → コミット → プッシュ。**
2. **コミット文は日本語で簡潔に。**
   　例：
   　`
   　fix: ConfigLoaderのファイルパス読み込み不具合を修正  
   　refactor: デルタ解析モジュールを分割し責務を整理  
   　`
3. **リモートホストへのプッシュを忘れない。**
   　`
   　git add .
   　git commit -m "〇〇を修正"
   　git push origin feature/〇〇
   　`

## 6. Issue 対応と報告ルール

1. **完了報告時は、Issue コメントを投稿する。**
   　- `.github/ISSUE_TEMPLATE/` にあるテンプレートを使用：
   　  - `bug.yml`（バグ修正）
   　  - `feature.yml`（新機能）
   　  - `refactor.yml`（リファクタリング）
2. **長文投稿はテンポラリファイルに出力してから投稿。**
   　- 投稿失敗防止のため、`temp_issue_comment.md` などを作成して投稿する。
   　- Windows PowerShell（`pwsh`）では、日本語の長文や全角記号を含む `gh issue comment -b` 直書きで PSReadLine がクラッシュする既知事象があるため、**必ず `-F <ファイル>` で投稿すること**。
   　- 文字コードは **UTF-8（BOMなし）** を推奨。改行はLF/CRLFいずれでも可（GitHub側で許容）。

    安全な投稿手順（PowerShell）:
    ```pwsh
    # 1) 本文をUTF-8で保存（例: temp_issue_comment.md）
    # 2) ファイル指定で投稿（絶対パス推奨）
    gh issue comment 62 -F "C:\MyProjects\ComfyMotionAnimator\temp_issue_comment.md"
    ```

    投稿確認（2通り）:
    ```pwsh
    # A) テキスト出力 + キーワード検索
    gh issue view 62 --comments | Out-String | Select-String -Pattern "完了報告|進捗報告"

    # B) API経由で直近数件を確認（URL/冒頭表示）
    gh api repos/EtoileArtStudio/ComfyMotionAnimator/issues/62/comments --paginate \
       --jq ".[ -3: ] | map({created_at, user: .user.login, html_url, head: (.body | .[0:30])})"
    ```

    取得時のコツ（機械可読にする）:
    ```pwsh
    # JSONで全コメントを取得
    gh issue view 62 --comments --json comments \
       --jq ".comments | map({author: .author.login, createdAt: .createdAt, body: .body})"
    ```

    トラブルシュート:
    - `-b` 直書きで例外（`PSReadLine IndexOutOfRangeException`等）が出る → **`-F` に切替**
    - それでも不安定な場合 → セッション限定で `Remove-Module PSReadLine` または `pwsh -NoProfile` を使用
    - 出力が途中で途切れる → 一時的に `$env:GH_PAGER = ''` を設定しページャ無効化
3. **コメント内容に含めるべき項目：**
   　- コーディング担当であることをまず記載する。”コーディング担当です。”など。
   　- 作業の目的と対応内容
   　- 修正／改善の要点
   　- レビューでの指摘と学び
   　- 今後の改善方針や反省点
     - レビュー依頼（レビューをお願いします。）や、確認事項など。

## 10. 教訓集

### 10.1 設計・実装フェーズ

1. **設計を先に文書化してからコードを書く**
   - 命名ルール・メタデータ構造は、実装前にIssueコメントなどで明文化する
   - 「設計どおりに実装する」を徹底し、設計メモを忘却しない
   - 文書化した設計は、テストケース設計の基盤にもなる
   - **問題点だけでなく「将来の方向性」も書く**（次のリファクタの方向が明確になる）
   - **責務境界は特定機能だけでなく、他機能にも適用表を作る**（拡張時の迷子防止）

2. **Enumベースのパターン分類を活用する**
   - 複数の分岐条件がある処理は、Enumで状態を明示的に分類する
   - 例：`OutputType.SINGLE / BATCH_VARIATION / MULTI_NODE`
   - 分類ロジックと処理ロジックを分離し、可読性を向上させる

3. **異常系・フォールバックを最初から考慮する**
   - 「正常系が動いた」で満足しない
   - `None`・空・構造不正など、入力の異常パターンを洗い出す
   - フォールバック処理とログ出力をセットで実装する

4. **ファイル操作などの「重い処理」にはロールバック方針を設計段階で決める**
   - 複数ステップの操作（ファイルリネーム＋メタデータ更新など）は、途中失敗時の対処を先に決める
   - 例：「ファイル操作が成功した場合のみメタデータを確定保存」
   - ロールバック方針がないと、中途半端な状態からの復旧が困難になる

5. **責務の集中に注意する**
   - 1つのクラスに機能を盛り込みすぎない
   - 「メソッドを増やす前に、既存の責務とのバランス」を確認する
   - 早めにモジュール分割・クラス抽出を検討する

### 10.2 テストフェーズ

6. **テストスケルトンを先に作成する**
   - 実装前に「何をテストすべきか」を明確化する
   - `pytest.skip()` でスケルトンを作り、実装後に有効化する
   - テストケースは設計メモから導出する

7. **テストは「失敗させてから成功させる」**
   - 最初から通るテストは検証力が弱い
   - 意図的に失敗させ、実装で成功に変えるサイクルを回す

8. **「一回限り」ではなく「サイクル」を想定したテストを書く**
   - 操作→戻す→再操作のサイクルが正しく動くかをテストする
   - 例：昇格→再昇格→元に戻す、のような繰り返し操作
   - 状態遷移の履歴（`promoted_from`等）が正しく保持されるか検証する

### 10.3 レビュー対応フェーズ

9. **レビュー指摘は即座に対応、報告は詳細に**
   - 指摘事項を一覧化し、対応状況を明示する
   - 「何を」「どう」修正したかをコミットメッセージとIssueコメントに記録する

10. **振り返りメモを残す習慣をつける**
   - 「どこが一番苦労したか」
   - 「次に改善したいポイントはどこか」
   - これらを記録することで、同じ苦労を繰り返さない

### 10.4 バグ発見時の心得

11. **参照先の確認を怠らない**
   - `config` vs `workflow_config` のような類似名の混同に注意
   - 変数やプロパティの「実際の中身」をデバッグログで確認する
   - 「動いているように見える」と「正しく動いている」は別物

### 10.5 ネットワーク通信・外部API連携

12. **HTTP通信には必ずtimeoutを設定する**
   - 「動くからいいや」で済ませず、ネットワーク異常時の挙動まで気を配る
   - 他の箇所でtimeoutを設定しているなら、同様の箇所も揃える（一貫性）
   - 例：`requests.get(url, timeout=5)` - ハング耐性を確保する

13. **戻り値がNoneになる条件をdocstringに明記する**
   - コードを読まないと分からない仕様は避ける
   - 「APIエラー時」「監視中の例外時」など、Noneを返す条件を列挙する
   - 呼び出し側が判断しやすいインターフェースを意識する

14. **将来の連携を見据えた導線を用意する**
   - 次に実装する機能との繋がりを意識したログ出力やコメントを残す
   - 例：`prompt_id`を返すなら、それを使う`fetch-status`の存在を示唆するログ
   - 「今の実装」と「次の一歩」を繋ぐ意識を持つ
