# AI Job Insight 基本仕様書

## 1. 目的
本書は「AI Job Insight」の基本仕様を定義することを目的とする。

本書では主に以下を定義する。

- システム全体の機能ブロック構成
- 機能ブロック間の接続関係
- 画面構成および画面項目の役割
- 基本的なデータの扱い

APIの呼び出し方法や具体的なプログラム構造などの実装詳細については、詳細設計で定義する。

---

# 2. システム概要

AI Job Insight は、ユーザーが入力した

- 案件情報
- 自身のプロフィール

をもとに、AIを利用して案件応募の適合度や応募戦略を分析する支援ツールである。

本システムは以下の特徴を持つ。

- Webブラウザ上で動作する
- AI APIをクライアントから直接呼び出す
- サーバー側でAI処理は行わない
- データ保存はブラウザ内ストレージを基本とする

---

# 3. システム前提

## 3.1 実行環境

- Webブラウザ

## 3.2 AI利用方式

### v1における方式
AI分析は外部AIサービスのAPIを利用して実行する。

ユーザーは自身のAPIキーを入力し、そのAPIキーを用いて分析を実行する。

### v2以降の方式
v1における方式に加えて、Googleログイン(OAuth)を利用したバックエンド通信によるAI利用に対応。

主な役割

- 分析ボタン押下の受付
- 入力データ確認
- AI分析処理の開始
- 分析結果の受信処理

## 4.3 AI応答サービスブロック
AI APIと通信し、分析を実行する機能。

v1ではフロントエンド側に処理を実装する。
v2以降では、フロントエンド側と、バックエンド側それぞれに処理を実装する。

### AI通信処理ブロック
AIとの通信処理を行う。

### AI応答整形ブロック
AIから取得した結果を、画面表示および履歴保存に適した形式へ整形する機能。

**Note**: 本ブロックは将来的に共通基盤機能として独立化される予定（詳細は「4.7 共通基盤機能ブロック」参照）。

## 4.4 結果表示系

### 分析結果表示ブロック
AI分析結果を画面へ表示する機能。

表示項目

- 応募おすすめ度
- 案件評価理由
- スキル適合度
- 応募戦略

### スキル適合度可視化機能

スキル適合度は、事前に設定されたユーザーのスキル項目を軸として可視化できるようにする。

表示方式は **レーダーチャート** を基本とする。

レーダーチャートでは、登録済みのスキル項目ごとに、対象案件に対する反応度または適合度を表示する。

スキル項目例

- 文書作成
- Web構築
- プログラミング
- 仕様検討
- イラスト作成

表示の考え方

- ホームページ制作やコーディングを含む案件では、Web構築やプログラミングの値を高く表示する
- 仕様が既に定まっている案件では、仕様検討の値は相対的に低く表示する
- 画像制作要求がない案件では、イラスト作成の値は 0 または極めて低い値とする

役割

- ユーザーの保有スキルのうち、どの要素が案件に対して有効に作用するかを視覚的に把握できるようにする
- 単なる文章説明だけでなく、案件との相性を図形的に示すことで直感的な理解を支援する
- ポートフォリオ的な見栄えやアピール性を高める

### 提案支援表示ブロック
応募支援およびプロフィール改善のアドバイスを表示する機能。

### 対話支援ブロック
案件分析結果の表示後に、ユーザーがAIと対話しながら追加検討や投稿文作成を行うための機能。

主な機能

- チャット欄の表示
- 対話モードの切替
- 追加情報の入力
- AIとの継続的なやり取り
- 投稿文の生成およびブラッシュアップ
- 作成結果の保存

対話モードは以下の2種類を持つ。

- 質問・相談モード
- 投稿文作成モード

#### 質問・相談モード

案件内容、分析結果、応募方針、注意点などについて、ユーザーがAIへ質問・相談できるモードとする。

#### 投稿文作成モード

ユーザーが追加情報を入力し、AIと対話しながら最終的な投稿文を作成・改善していくモードとする。

追加情報の例

- 既にエントリーしている人数
- 他応募者の投稿傾向
- コンペ案件における他候補の特徴
- ユーザー側で新たに判明した条件や補足情報

本モードでは、単なるチャット応答ではなく、投稿文のたたき台を生成し、ユーザーとのやり取りを通じて文章を段階的に改善できる構成とする。

また、最終的に整理された投稿文は保存できるようにする。

### 応答促進メッセージ表示

基本分析結果の末尾には、追加対話を促す案内を表示する。

案内文の例

- 質問はありますか？
- 追加で情報をいただければ、一緒に投稿文を作成することもできます。

役割

- 分析結果の閲覧だけで終わらせず、次のユーザー行動につなげる
- AIとの継続対話へ自然に誘導する
- 投稿文作成支援機能の利用率を高める

## 4.7 共通基盤機能ブロック（2026年3月15日追加）

将来的に別プロダクト（Shopping Insight等）への横展開を見据え、再利用可能な共通機能を独立したブロックとして定義する。

本ブロックは、AI Job Insightリポジトリ内で将来の切り出しを想定した設計として整備される。

### 設計原則

1. **Job固有の概念を含まない**
   - 「案件」「プロフィール」などの固有用語を排除
   - 汎用的な概念のみで設計

2. **依存方向の明確化**
   - 共通基盤機能はJob固有機能に依存しない
   - Job固有機能は共通基盤機能を利用する

3. **拡張性の確保**
   - データ構造は拡張可能な設計
   - 新たなプロダクトでの再利用を想定

### 4.7.1 共通AI応答基盤機能

AI APIとの通信および応答整形を行う汎用的な機能。

#### AI通信処理
**責務**
- AIサービスとのHTTPS通信
- リクエスト/レスポンス管理
- エラーハンドリング
- タイムアウト管理

**対象AIサービス**
- OpenAI API
- Google Gemini API

**入力（汎用パラメータ）**
- service: AIサービス種別
- apiKey: APIキー
- prompt: プロンプト文
- modelName: モデル名
- temperature: 生成パラメータ

**出力**
- rawResponse: 生レスポンス
- error: エラー情報

#### AI応答整形処理
**責務**
- 生レスポンスの正規化
- JSON形式への変換
- 構造化データへの変換
- 汎用データ形式への変換

**入力**
- rawResponse: 生レスポンステキスト
- responseFormat: 応答形式

**出力（汎用構造）**
```typescript
interface GenericAIResponse {
  content: string;
  structuredData?: Record<string, any>;
  parseError?: string;
}
```

### 4.7.2 共通可視化機能

分析結果を視覚的に表示するための汎用コンポーネント群。

#### 星評価表示コンポーネント
**責務**
- 数値スコアの星表示への変換
- 星アイコンの描画
- アクセシビリティ対応

**入力プロパティ**
- score: スコア値（0-5）
- maxStars: 最大星数
- size: 表示サイズ
- readonly: 読み取り専用フラグ

#### レーダーチャートコンポーネント
**責務**
- 多軸データの可視化
- レーダーチャートの描画
- ラベル・値の表示

**入力データ形式**
```typescript
interface RadarChartData {
  labels: string[];    // 軸ラベル
  values: number[];    // 各軸の値
  maxValue?: number;   // 最大値
}
```

#### 結果カード表示コンポーネント
**責務**
- 汎用的な結果表示カードの提供
- タイトル・説明・スコアの表示
- レスポンシブ対応

**入力プロパティ**
- title: タイトル
- description: 説明文
- score: スコア
- variant: 表示バリエーション

### 4.7.3 共通データ構造

分析結果を汎用的に扱うためのデータ構造定義。

```typescript
interface GenericAnalysisResult {
  score: number;                    // 総合スコア（0-100）
  summary: string;                  // 概要
  strengths: string[];              // 強み
  concerns: string[];               // 懸念点
  strategy: string;                 // 戦略・推奨事項
  skillMatch?: {                    // スキルマッチング
    labels: string[];
    values: number[];
  };
  metadata?: Record<string, any>;   // 拡張データ
}
```

### 4.7.4 Job固有機能との関係

Job固有機能（プロンプト定義、評価軸定義、画面導線など）は、共通基盤機能を組み合わせて利用する。

**依存方向**
```
Job固有機能 → 共通基盤機能
```

**例：案件分析の流れ**
1. Job固有：案件分析プロンプト生成
2. 共通：AI通信処理でプロンプト送信
3. 共通：AI応答整形処理で構造化
4. Job固有：Job用評価軸でデータ解釈
5. 共通：可視化コンポーネントで表示

### 4.7.5 将来の横展開イメージ

**Shopping Insightへの展開例**
- Shopping固有：商品分析プロンプト定義
- Shopping固有：商品評価軸定義
- 共通：AI通信・応答整形（再利用）
- 共通：可視化コンポーネント（再利用）

## 7.4 プロフィール入力領域

本領域では、案件分析に使用するためのプロフィール情報を入力する。

入力情報は **分析用プロフィール情報** として保持され、案件との適合度分析に利用される。

また、本システムではクラウドワークス等のプロフィールに貼り付け可能な文章を生成するため、分析用情報とは別に **生成プロフィール文** を扱う。

### 分析用プロフィール入力項目

以下の項目を入力する。

- 自己紹介
- スキル
- 実績
- 得意分野

また、スキル適合度の可視化に用いるため、必要に応じて **スキル項目ラベル** を設定できるようにする。

スキル項目ラベル例

- 文書作成
- Web構築
- プログラミング
- 仕様検討
- イラスト作成

役割

案件との適合度分析において、応募者側の情報としてAI分析の入力データに使用される。

また、スキル項目ラベルはスキル適合度のレーダーチャート表示における軸情報として使用される。

### プロフィール文生成機能

プロフィール入力領域には **プロフィール文作成ボタン** を設ける。

役割

入力されたプロフィール情報をもとに、AIがクラウドワークス等のプロフィールページに掲載可能な文章形式のプロフィール文を生成する。

生成されたプロフィール文は以下の用途で使用する。

- クラウドワークスのプロフィール欄への貼り付け
- 応募文作成時の基礎情報

なお、プロフィール文の新規生成または既存プロフィール文の書き換え提案を行う場合も、AI生成結果の保存反映は「提案提示 → ユーザー確認 → 承認後反映」の共通ルールに従うものとする。

### プロフィール文文字数設定

プロフィール文生成時には、生成する文章の文字数上限を指定できるようにする。

初期設定は **1000文字以内** とする。

これは、クラウドワークスのプロフィール文字数制限が 1024 文字であることを考慮し、貼り付け時の余裕を確保するためである。

ユーザーは必要に応じて文字数上限を変更できるものとする。

役割

- 掲載先サービスの文字数制限に合わせてプロフィール文の長さを調整する
- デフォルトではクラウドワークス向けの使いやすさを確保する

### 生成プロフィール文表示

AIにより生成されたプロフィール文を表示する専用表示領域を設ける。

ユーザーはこの文章をコピーして外部サービスへ貼り付けることができる。

表示時には、必要に応じて文字数上限との関係を確認できるよう、生成結果の文字量を把握しやすい構成とする。

### 分析時の参照データ

案件分析時には

- 分析用プロフィール入力項目

を参照する。

生成されたプロフィール文は分析処理では使用しない。

## 7.9 対話支援領域

案件についての分析結果表示の下部には、AIと対話できるチャット領域を設ける。

本領域では、対話モードを切り替えて利用できるものとする。

対話モード

- 質問・相談モード
- 投稿文作成モード

### 質問・相談モード

案件内容や分析結果について、ユーザーが自由に質問・相談できるモードとする。

### 投稿文作成モード

ユーザーが追加情報を入力し、AIと対話しながら投稿文を段階的に作成・改善していくモードとする。

入力できる追加情報の例

- 既にエントリーしている人数
- 他応募者の投稿内容や傾向
- コンペ案件における比較情報
- ユーザーが後から取得した補足条件

本モードでは、単発の回答ではなく、作成中の投稿文を継続的にブラッシュアップできる構成とする。

また、最終的に整理された投稿文は保存できるようにする。

### 投稿文作成方式選択

投稿文作成モードでは、投稿文の作成方法を選択できるようにする。

選択肢

- 既存テンプレートを使用して作成する
- 一から新規作成する

既存テンプレートを使用する場合は、テンプレート内容をもとに必要情報を埋める形で投稿文を作成できるようにする。

### テンプレート化機能

作成した投稿文は、再利用可能なテンプレートとして保存できるようにする。

テンプレート保存時には、以下の方法を選択できるようにする。

- AIによる成形処理を行ってテンプレート化する
- ユーザーが手作業で編集した内容をそのままテンプレート化する

テンプレート保存前には、AI整形を利用するか、手作業で整えるかをユーザーが選択できるものとする。

### AI提案反映方式

本システムでは、AIが生成または改善提案した結果を保存対象データへ反映する際は、**提案提示 → ユーザー確認 → 承認後反映** を共通ルールとする。

本ルールは、投稿文作成時に限らず、AIが生成結果を保存対象へ反映するすべての場面に適用する。

適用対象例

- 投稿文の新規作成・改善
- テンプレートのブラッシュアップ
- プロフィール文の新規生成
- 既存プロフィール文の書き換え提案
- その他、AIが保存対象データの更新案を提示する処理

反映方式

- AIは保存対象データを自動で直接書き換えない
- AIは提案内容をユーザーへ提示する
- ユーザーが内容を確認し、承認した場合にのみ本体データへ反映する
- 承認前の提案は、本体データとは分離して保持できるものとする

役割

- ユーザーが意図しない書き換えを防止する
- AIによる改善提案を確認しながら、段階的に内容を完成させる
- データの主導権を常にユーザー側へ残す
- AI生成結果の保存に関する挙動を全機能で統一する

### 保存機能

投稿文作成モードで作成・改善した結果は、後で再利用できるよう保存対象とする。

# 8. データ仕様

## 8.1 保存データ

本システムでは、ユーザーが入力・生成・設定した情報のうち、再利用性があるものを永続化対象とする。

保存対象

- APIキー
- 分析用プロフィール情報
  - 自己紹介
  - スキル
  - 実績
  - 得意分野
  - スキル項目ラベル
- 生成プロフィール文
- プロフィール文の文字数上限設定
- 案件履歴（案件内容/案件URL/案件メモ)
- 分析日時
- 投稿文作成モードで保存した投稿文
- テンプレート化した投稿文
- 対話支援機能において保存対象とした作成結果
- 応募予定案件リスト
- テンプレート本体データ
- AIが提示した未承認の改善提案データ
- ユーザーが変更した各種設定値

保存方針

- アプリ利用中に作成されたデータ、およびユーザー操作により変更された設定は、再利用できるよう永続化する
- 次回アクセス時にも前回の状態を可能な範囲で復元できるようにする
- 永続化対象のうち、案件分析の一時入力値については、必要に応じて保存対象とする
- 永続化されたデータは、ブラウザ機能に依存せず、アプリ内操作でも削除または初期化できるようにする

保存場所

ブラウザストレージ
